From pi-subagents package to available subagent tool

First internal map: how Pi discovers the package, loads the extension, calls its factory, and ends up with a tool the parent agent can choose.

package loadingextension factorytool registrationschemaparent agent

One-sentence model

Pi reads the pi manifest in package.json, loads src/extension/index.ts, calls its default export with ExtensionAPI, and that extension calls pi.registerTool({ name: "subagent", ... }).

The discovery timeline

Installation records the package source.
User-facing install is pi install npm:pi-subagents. Pi package docs say npm, git, and local package sources can be installed and then loaded by Pi.
Pi inspects package resources.
The manifest says: 
{
  "pi": {
    "extensions": ["./src/extension/index.ts"],
    "skills": ["./skills"],
    "prompts": ["./prompts"]
  }
}
That means this package contributes one extension entrypoint plus skill and prompt resources.
Pi loads the TypeScript extension.
Pi extension docs state extensions are TypeScript modules loaded through jiti, so no build step is required for index.ts.
Pi calls the default exported factory.
In this repo, the export is export default function registerSubagentExtension(pi: ExtensionAPI): void. The pi object is the extension's access to Pi internals.
The extension avoids recursively registering in ordinary child agents.
At the top of the factory, it checks SUBAGENT_CHILD_ENV. If this Pi process is a child subagent, it normally returns early. This prevents every child from automatically exposing another full subagent tool unless special fanout-child mode is active.
The extension initializes runtime state.
It creates result/async directories, loads config, starts result watchers, creates async job tracking, and builds a foreground executor with createSubagentExecutor(...).
The extension builds the tool definition.
The local variable tool has name: "subagent", a long description, parameters: SubagentParams, an execute(...) function, and TUI renderers.
The extension registers the tool.
pi.registerTool(tool) is the decisive line. After this, Pi has a custom tool named subagent in its tool registry for the parent agent.
The model sees the tool contract.
When tools are active, Pi can serialize the tool name, description, and parameter schema into the provider request. The parent model can then emit a tool call like { action: "list" } or { agent: "reviewer", task: "..." }.

The key source landmarks

// package.json
"pi": {
  "extensions": ["./src/extension/index.ts"],
  "skills": ["./skills"],
  "prompts": ["./prompts"]
}
// src/extension/index.ts
export default function registerSubagentExtension(pi: ExtensionAPI): void {
  ...
  const tool: ToolDefinition<typeof SubagentParams, Details> = {
    name: "subagent",
    parameters: SubagentParams,
    execute(id, params, signal, onUpdate, ctx) {
      return executeSubagentCollapsed(id, params, signal, onUpdate, ctx);
    },
    ...
  };

  pi.registerTool(tool);
}
// src/extension/schemas.ts
export const SubagentParams = Type.Object({
  agent: Type.Optional(Type.String(...)),
  action: Type.Optional(Type.String({ enum: [...SUBAGENT_ACTIONS] })),
  tasks: Type.Optional(Type.Array(TaskItem, ...)),
  chain: Type.Optional(Type.Array(ChainItem, ...)),
  context: Type.Optional(Type.String({ enum: ["fresh", "fork"] })),
  async: Type.Optional(Type.Boolean(...)),
  ...
});

What is not happening

  • Installing pi-subagents does not start background reviewers automatically. The README explicitly says it gives Pi a delegation tool.
  • The builtin agents are not themselves the tool. They are discoverable configurations/prompts that the subagent tool can execute.
  • The package manifest does not execute work. It only tells Pi which resources to load.
